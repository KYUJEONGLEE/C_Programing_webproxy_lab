## 0. 이 자료를 어떻게 쓰면 좋은가

이 자료는 **"개념을 다 읽고 구현"** 방식보다,  
**"지금 만들 것에 필요한 만큼 이해 -> 바로 구현 -> 테스트 -> 복습 기록"** 흐름에 맞춰 만들어졌다.

권장 사용 순서는 아래와 같다.

1. **세션 시작 전**: 이번 세션에 해당하는 개념 파트를 읽는다.
2. **구현 직전**: 시스템콜 흐름과 체크리스트를 다시 본다.
3. **구현 중 막힐 때**: "자주 헷갈리는 지점"을 먼저 확인한다.
4. **구현 후**: 셀프 테스트 질문에 답해 본다.
5. **세션 종료 직전**: 별도 템플릿 파일에 복습 노트를 작성한다.

이 자료의 핵심은 세 가지다.

- **흐름으로 이해하기**: 함수 하나씩 외우기보다, "요청 하나가 어디서 어디로 흐르는지"를 이해한다.
- **역할로 구분하기**: server fd, connected fd, client fd를 헷갈리지 않는다.
- **테스트로 확인하기**: 구현보다 테스트가 이해를 더 빠르게 만든다.

## 1. 전체 로드맵 한눈에 보기

### 최종 도착점
소켓 이해 -> echo 서버 -> tiny 웹서버 -> 숙제 3문제 이상 -> proxy 서버

### 왜 이 순서인가
- **echo**는 소켓 인터페이스를 손으로 만져 보는 가장 작은 서버다.
- **tiny**는 echo 위에 HTTP, 파일 전송, CGI, 프로세스 제어를 얹는다.
- **proxy**는 tiny까지의 내용을 전부 재사용하면서 동시성과 캐시가 추가된다.

### 의존 관계
- 네트워크/소켓 이해가 없으면 echo가 안 잡힌다.
- echo를 모르면 tiny에서 descriptor 흐름이 흔들린다.
- tiny의 HTTP를 모르면 proxy에서 요청 파싱/재작성에 막힌다.
- 동시성 개념이 없으면 proxy 캐시가 불안정해진다.

## 2-1. 네트워크 기본 개념

### 1) 네트워크 프로그래밍에서 진짜로 중요한 것
네트워크 프로그래밍은 "멀리 있는 프로세스와 파일 디스크립터처럼 통신하는 방법"이라고 보면 된다.  
핵심은 세 가지다.

- **주소가 있어야 한다**: 누구에게 보낼지 알아야 한다. (IP, DNS)
- **포트가 있어야 한다**: 그 호스트 안의 어느 프로세스와 이야기할지 알아야 한다.
- **규약이 있어야 한다**: 데이터를 어떤 방식으로 주고받을지 정해야 한다. (TCP/UDP/HTTP)

### 2) TCP/IP 4계층
#### 애플리케이션 계층
HTTP, DNS 같은 "무슨 말을 할지"를 정의한다.

#### 전송 계층
TCP/UDP가 있다.

- **TCP**
  - 연결 지향
  - 바이트 스트림
  - 순서 보장
  - 신뢰성 있음
- **UDP**
  - 비연결성
  - 데이터그램 단위
  - 순서/재전송 보장 없음
  - 단순하고 빠름

#### 인터넷 계층
IP 주소를 기반으로 패킷을 어디로 보낼지 결정한다.

#### 링크 계층
같은 네트워크 구간에서 실제 프레임을 전송한다.

### 3) IP 주소와 포트
- **IP 주소**: 호스트를 찾는 주소
- **포트 번호**: 그 호스트 내부의 특정 프로세스/서비스를 찾는 번호

예를 들어 `203.0.113.10:8080`은  
"203.0.113.10이라는 호스트의 8080 포트로 접속"을 뜻한다.

### 4) DNS
사람은 도메인을 기억하고, 네트워크는 IP 주소로 동작한다.  
DNS는 **도메인 이름을 IP 주소로 바꾸는 시스템**이다.

`getaddrinfo()`는 이 과정을 포함해서  
도메인/포트를 실제 연결 가능한 주소 정보 목록으로 바꿔 준다.

### 5) 바이트 순서
CPU 아키텍처마다 정수의 메모리 저장 순서가 다를 수 있다.

- **Host byte order**: 내 컴퓨터가 쓰는 순서
- **Network byte order**: 네트워크에서 약속한 순서 (big endian)

그래서 정수형 포트/주소 일부는 변환이 필요하다.

- `htons`: host to network short
- `htonl`: host to network long
- `ntohs`: network to host short
- `ntohl`: network to host long

### 꼭 잡아야 할 문장
> 네트워크 통신은 "IP로 호스트를 찾고, 포트로 프로세스를 찾고, TCP/UDP 규약으로 데이터를 주고받는 것"이다.

### 자주 헷갈리는 지점
- IP 주소와 포트를 같은 것으로 생각하는 실수
- DNS를 "프로토콜"만으로 외우고 실제 역할을 놓치는 실수
- 바이트 순서를 "문자열 변환"으로 오해하는 실수

### 셀프 체크
1. IP 주소와 포트 번호는 각각 무엇을 식별하는가?
2. TCP와 UDP의 가장 큰 차이는 무엇인가?
3. 왜 네트워크 바이트 순서가 필요한가?

## 2-2. 소켓 기본 개념

### 1) 소켓이란
소켓은 **통신을 위한 파일 디스크립터**다.  
네트워크도 결국 read/write/close로 다룰 수 있게 해 주는 운영체제 인터페이스라고 보면 된다.

즉,
- 파일은 디스크에 있는 데이터에 대한 fd
- 소켓은 네트워크 연결에 대한 fd

### 2) 파일 디스크립터 관점이 중요한 이유
처음 배우면 소켓을 "특별한 것"처럼 느끼기 쉽다.  
그런데 서버 코드를 읽을 때는 이렇게 생각해야 한다.

- **listenfd**: 손님을 받는 문
- **connfd**: 실제 손님과 대화하는 창구

문과 대화 창구는 같은 것이 아니다.

### 3) TCP 소켓 vs UDP 소켓
#### Stream Socket (TCP)
- 연결 후 통신
- 바이트 스트림
- 순서/신뢰성 보장
- echo, tiny, proxy에서 주력으로 사용

#### Datagram Socket (UDP)
- 연결 없이 메시지 단위 전송
- 데이터 경계 유지
- 손실/순서 변경 가능

### 4) 소켓 주소 구조체
소켓은 결국 "어디와 통신할지"를 알아야 해서 주소 구조체가 필요하다.

대표적으로:
- `sockaddr`
- `sockaddr_in` (IPv4)
- `sockaddr_in6` (IPv6)
- `sockaddr_storage` (충분히 큰 범용 저장용)

현대적인 코드에서는 직접 구조체를 만지기보다 `getaddrinfo()` 결과를 쓰는 경우가 많다.

### 꼭 잡아야 할 문장
> 소켓은 네트워크 통신을 파일 디스크립터처럼 다루기 위한 OS 인터페이스다.

### 자주 헷갈리는 지점
- listenfd와 connfd를 같은 fd로 생각하는 실수
- server가 받은 connfd로 다음 클라이언트까지 계속 accept한다고 오해하는 실수
- TCP가 "패킷 단위"로 read되는 줄 아는 실수

### 셀프 체크
1. 왜 소켓을 파일 디스크립터처럼 생각하면 코드가 쉬워지는가?
2. listenfd와 connfd는 역할이 어떻게 다른가?
3. TCP는 왜 "메시지"보다 "바이트 스트림"으로 이해해야 하는가?

## 2-3. 소켓 인터페이스

### 1) 서버 쪽 큰 흐름
서버는 대체로 아래 순서를 따른다.

1. `socket()` - 소켓 생성
2. `bind()` - 내 주소/포트에 묶기
3. `listen()` - 연결 요청을 받을 준비
4. `accept()` - 실제 연결 하나 수락
5. `read/write` 또는 RIO - 데이터 송수신
6. `close()` - 연결 종료

### 2) 클라이언트 쪽 큰 흐름
클라이언트는 대체로 아래 순서를 따른다.

1. `socket()` - 소켓 생성
2. `connect()` - 서버 주소로 연결 시도
3. `read/write` 또는 RIO - 데이터 송수신
4. `close()` - 종료

### 3) 왜 서버는 bind가 필요하고 클라이언트는 보통 필요 없나
서버는 **"이 포트에서 기다리겠다"**고 공개적으로 선언해야 한다.  
그래서 `bind()`가 필요하다.

반면 클라이언트는 접속을 시작하는 쪽이기 때문에  
보통 로컬 포트를 OS가 자동으로 골라 준다.  
그래서 일반적인 클라이언트 코드는 `bind()`를 직접 하지 않는다.

### 4) `listen()`의 의미
`listen()`은 "소켓 생성"이 아니라  
**그 소켓을 수신 대기 소켓으로 바꾸는 단계**다.

### 5) `accept()`의 의미
`accept()`는 listenfd 자체를 반환하는 게 아니다.  
새 연결마다 **새로운 connected descriptor**를 만들어 준다.

이게 정말 중요하다.

- listenfd: 계속 살아 있음
- connfd: 클라이언트 1명과의 대화용

### 6) `getaddrinfo()` / `getnameinfo()`
#### `getaddrinfo()`
사람이 넣은 host/service를 실제 연결 가능한 주소 정보 리스트로 바꾼다.  
클라이언트/서버 양쪽 모두에 유용하다.

#### `getnameinfo()`
주소 구조체를 사람이 읽을 수 있는 host/service 문자열로 바꾼다.

### 7) `open_clientfd()`, `open_listenfd()`
CS:APP에서는 자주 쓰는 반복 패턴을 감춘 래퍼 함수를 쓴다.

- `open_clientfd(host, port)`
  - getaddrinfo -> socket -> connect 반복 시도
- `open_listenfd(port)`
  - getaddrinfo -> socket -> bind -> listen

학습 포인트는 **래퍼 내부 흐름을 직접 설명할 수 있어야 한다**는 점이다.

### 암기용 흐름
#### 서버
`socket -> bind -> listen -> accept -> read/write -> close(connfd)`

#### 클라이언트
`socket -> connect -> read/write -> close`

### 자주 헷갈리는 지점
- `accept()` 전에 클라이언트와 통신할 수 있다고 생각하는 실수
- `listenfd`에 직접 데이터를 쓰는 실수
- `close(connfd)`를 해 놓고 listenfd까지 종료해 버리는 실수

### 셀프 체크
1. 왜 `accept()`는 새 fd를 반환하는가?
2. `listen()`은 정확히 무엇을 준비하는 단계인가?
3. `open_listenfd()` 내부에 어떤 시스템콜들이 들어갈지 말로 설명할 수 있는가?

## 2-4. 견고한 I/O (RIO)

### 1) 왜 그냥 read/write만 쓰면 불안한가
네트워크에서는 한 번의 `read()`가  
원하는 길이만큼 항상 다 읽어 준다는 보장이 없다.  
이걸 **short count**라고 한다.

예를 들어 100바이트를 읽고 싶어도
- 30바이트만 들어올 수 있고
- 나머지는 다음 호출에서 들어올 수 있다.

### 2) RIO가 해결하는 것
CS:APP의 RIO는 이런 문제를 덜어 주는 래퍼다.

- `rio_readn`: n바이트를 최대한 끝까지 읽음
- `rio_writen`: n바이트를 끝까지 씀
- `rio_readlineb`: 줄 단위로 읽음
- 내부 버퍼를 사용해 효율적으로 읽음

### 3) 언제 무엇을 쓰나
- **고정 길이 데이터**를 다 읽고 싶을 때: `rio_readn`
- **한 줄씩 헤더를 읽고 싶을 때**: `rio_readlineb`
- **응답을 안정적으로 보내고 싶을 때**: `rio_writen`

### 4) tiny/proxy에서 왜 중요해지나
HTTP 헤더는 줄 단위다.  
그래서 `rio_readlineb()`로 요청 라인과 헤더를 읽는 패턴이 매우 자연스럽다.

### 꼭 잡아야 할 문장
> RIO는 네트워크 I/O에서 "덜 읽히는 문제"와 "헤더를 줄 단위로 읽는 문제"를 다루기 쉽게 만든다.

### 자주 헷갈리는 지점
- `rio_readn`과 `rio_readlineb`의 목적을 섞는 실수
- short count가 에러라고 생각하는 실수
- 내부 버퍼가 있으니 아무 read나 섞어 써도 된다고 생각하는 실수

### 셀프 체크
1. short count는 왜 생길 수 있는가?
2. HTTP 헤더를 읽을 때 `rio_readlineb()`가 잘 맞는 이유는 무엇인가?
3. echo 서버에서 RIO를 쓰면 어떤 장점이 있는가?

## 2-5. 클라이언트-서버 모델

### 1) 요청-응답 흐름
가장 단순한 형태는 아래와 같다.

1. 클라이언트가 접속
2. 서버가 연결 수락
3. 클라이언트가 요청 전송
4. 서버가 처리 후 응답 전송
5. 연결 종료 또는 유지

echo는 이 구조의 가장 단순한 예시다.  
"받은 걸 그대로 돌려준다"는 점만 다르다.

### 2) Iterative server
iterative server는 **한 번에 한 클라이언트만 처리**한다.  
구조는 단순하지만 동시에 여러 요청이 오면 기다리게 된다.

echo, tiny 기본 버전, proxy 순차 버전에서 이 구조를 먼저 이해하면 좋다.

### 3) telnet 테스트가 좋은 이유
브라우저는 자동으로 많은 걸 숨긴다.  
반면 telnet은 사용자가 직접 한 줄씩 보내므로  
프로토콜과 소켓 동작을 눈으로 확인하기 좋다.

## 2-6. echo 서버 구현 학습자료

### 구현 전에 머릿속에 있어야 할 그림
- 서버는 listenfd를 만들고 무한 루프를 돈다.
- 매 루프에서 `accept()`로 connfd 하나를 만든다.
- `echo(connfd)`가 클라이언트가 보낸 바이트를 다시 돌려준다.
- 처리 후 connfd를 닫는다.
- listenfd는 계속 살아 있다.

### 구현 체크리스트
#### echo 클라이언트
- 서버 host/port로 접속한다.
- 표준입력에서 한 줄 읽는다.
- 서버에 보낸다.
- 서버 응답을 읽어서 출력한다.

#### echo 서버
- 포트를 받아 listenfd를 연다.
- 무한 루프에서 연결을 받는다.
- 받은 연결마다 echo 함수로 처리한다.
- 처리 후 connfd를 닫는다.

### 테스트 체크리스트
- telnet 또는 netcat으로 접속이 되는가
- 입력한 줄이 그대로 돌아오는가
- 여러 번 접속해도 서버가 살아 있는가
- 클라이언트 종료 후 서버가 다음 연결을 계속 받는가

### 자주 나는 버그
- 포트를 잘못 넘김
- listenfd 대신 connfd가 아닌 대상에 읽기/쓰기를 시도
- 개행 처리 때문에 결과가 다르게 보임
- `close()` 위치를 잘못 둬서 연결이 바로 끊김

### echo 단계에서 반드시 말할 수 있어야 할 것
1. 서버 쪽 시스템콜 순서
2. 클라이언트 쪽 시스템콜 순서
3. 왜 서버는 bind가 필요하고 클라이언트는 보통 필요 없는지
4. connfd가 왜 클라이언트마다 새로 생기는지

### echo_note에 남길 내용
- 서버와 클라이언트의 fd 흐름
- RIO가 왜 편했는지
- 내가 실제로 겪은 버그 2개 이상
- telnet 테스트에서 확인한 사실

## 3-1. HTTP 프로토콜

### 1) HTTP는 무엇인가
HTTP는 웹 클라이언트와 웹 서버가  
**어떤 형식으로 요청과 응답을 주고받을지**를 정한 애플리케이션 계층 프로토콜이다.

### 2) 요청 구조
대략 이렇게 생긴다.

- 요청 라인
- 헤더들
- 빈 줄
- 본문(있을 수도 있고 없을 수도 있음)

예시 형태:
- Request line: `GET /index.html HTTP/1.1`
- Header: `Host: example.com`
- Header: `User-Agent: ...`
- Blank line: `\r\n`

### 3) 응답 구조
- 상태 라인
- 헤더들
- 빈 줄
- 본문

예시 형태:
- Status line: `HTTP/1.0 200 OK`
- Header: `Content-length: ...`
- Header: `Content-type: text/html`
- Blank line
- Body

### 4) 메서드
- **GET**: 자원 요청
- **HEAD**: 헤더만 요청, 본문은 없음
- **POST**: 서버에 데이터 전달

tiny에서는 기본적으로 GET부터 이해하면 된다.  
숙제로 HEAD를 추가하면 HTTP 구조 이해가 훨씬 깊어진다.

### 5) 상태 코드
- `200 OK`: 성공
- `404 Not Found`: 파일/리소스 없음
- `403 Forbidden`: 접근 금지
- `501 Not Implemented`: 서버가 메서드를 지원하지 않음

### 6) HTTP/1.0 vs 1.1
#### HTTP/1.0
- 단순함
- 연결을 요청마다 끊는 모델이 흔함

#### HTTP/1.1
- Host 헤더 중요
- persistent connection 개념 강화
- 더 엄격한 규칙이 많음

tiny는 교육용 서버라 단순화된 HTTP/1.0 관점으로 이해해도 된다.  
다만 브라우저는 보통 1.1 요청을 보내므로 Host 헤더를 보게 된다.

### 7) Host 헤더가 필요한 이유
하나의 서버가 여러 도메인을 서비스할 수 있기 때문이다.  
같은 IP로 들어와도 어느 도메인 요청인지 구분해야 한다.

### 꼭 잡아야 할 문장
> tiny는 "HTTP 요청을 읽고, 정적이면 파일을 보내고, 동적이면 CGI를 실행해 결과를 돌려주는 서버"다.

### 자주 헷갈리는 지점
- 헤더 끝의 빈 줄을 빼먹는 실수
- 요청 라인과 헤더를 한 덩어리 문자열로만 외우는 실수
- HTTP 버전과 Host 헤더의 관계를 놓치는 실수

### 셀프 체크
1. HTTP 요청은 어떤 네 부분으로 나뉘는가?
2. HEAD와 GET의 차이는 무엇인가?
3. Host 헤더는 왜 필요한가?

## 3-2. 웹 콘텐츠 처리

### 1) MIME type
브라우저는 받은 데이터가 무엇인지 알아야 한다.  
그때 쓰는 것이 `Content-type` 헤더다.

예:
- `text/html`
- `text/plain`
- `image/gif`
- `image/jpeg`
- `video/mpeg`

서버는 파일 확장자 등을 바탕으로 적절한 MIME type을 정한다.

### 2) 정적 콘텐츠 vs 동적 콘텐츠
#### 정적 콘텐츠
디스크에 있는 파일을 그대로 읽어서 보낸다.
예: html, jpg, gif, mp4

#### 동적 콘텐츠
프로그램을 실행해서 결과를 생성한 뒤 보낸다.
예: CGI 프로그램의 출력

### 3) URI 파싱
tiny는 요청된 URI를 보고  
이게 정적인지 동적인지 구분해야 한다.

보통 교육용 tiny에서는
- `cgi-bin`이 포함되면 동적
- 아니면 정적

같은 단순 규칙을 사용한다.

### 4) CGI의 핵심
CGI는 웹 서버가 외부 프로그램을 실행하고  
그 프로그램의 표준출력을 HTTP 응답 본문처럼 클라이언트에게 넘기는 방식이다.

서버가 하는 일:
1. 요청에서 쿼리 문자열 분리
2. 환경변수 설정 (`QUERY_STRING` 등)
3. `fork()`
4. 자식에서 `dup2()`로 표준출력을 클라이언트 소켓에 연결
5. `execve()`로 CGI 실행

### 5) 환경변수 전달
CGI 프로그램은 보통 인자를 argv보다 환경변수로 받는다.

대표 예:
- `QUERY_STRING`
- `REQUEST_METHOD`

### 자주 헷갈리는 지점
- 정적/동적 구분이 "파일 확장자"라고만 생각하는 실수
- CGI가 서버 내부 함수 호출이라고 생각하는 실수
- `dup2()`가 왜 필요한지 모르는 상태로 외우는 실수

### 셀프 체크
1. 정적 콘텐츠와 동적 콘텐츠의 본질적인 차이는 무엇인가?
2. CGI 프로그램의 출력이 브라우저에 보이려면 왜 `dup2()`가 필요한가?
3. `QUERY_STRING`은 어디서 오는가?

## 3-3. 파일 전송 메커니즘

### 1) `stat()`
정적 파일을 보내기 전에 서버는 확인해야 한다.

- 파일이 존재하는가
- 일반 파일인가
- 읽기 권한이 있는가

이때 `stat()`가 쓰인다.

### 2) `mmap()`
`mmap()`은 파일을 메모리에 매핑해  
메모리처럼 접근할 수 있게 해 준다.

tiny에서는 정적 파일을 읽어 보낼 때  
간단하고 빠른 방법으로 자주 등장한다.

흐름은 대략:
1. 파일 열기
2. `mmap()`
3. 응답 헤더 전송
4. 파일 내용 전송
5. `munmap()`

### 3) `malloc + rio_readn` 대안
숙제 11.9는 `mmap()` 없이도 파일을 읽어 전송하는 방법을 생각하게 만든다.

흐름은 대략:
1. 파일 열기
2. 버퍼 할당
3. 파일을 읽음
4. 클라이언트에 씀
5. 버퍼 해제

핵심은 **기능은 같지만 구현 방식이 다르다**는 점이다.

### 셀프 체크

2. `mmap()`은 개념적으로 무엇을 하는가?
3. `malloc + read/write` 방식과 `mmap()` 방식은 어떤 면에서 다를까?

## 3-4. 프로세스와 시그널

### 1) `fork()` / `execve()`
- `fork()`: 현재 프로세스를 복제
- `execve()`: 현재 프로세스 이미지를 새 프로그램으로 교체

CGI에서는 대개
- 부모는 웹 서버 역할 유지
- 자식은 CGI 프로그램으로 바뀜

### 2) `dup2()`
파일 디스크립터를 다른 번호에 복제한다.  
CGI에서 핵심은:

- 자식 프로세스의 표준출력(1)을
- 클라이언트 소켓 fd로 바꾸는 것

그러면 CGI 프로그램이 `printf()`로 쓰는 내용이  
바로 클라이언트로 간다.

### 3) `wait()` / `waitpid()`
자식 프로세스가 끝나면 부모가 수거해야 한다.  
안 하면 **좀비 프로세스**가 남는다.

### 4) `SIGCHLD`
자식이 종료될 때 부모에게 오는 시그널이다.  
서버는 보통 이 시그널을 처리해 좀비를 정리한다.

### 5) `SIGPIPE` / `EPIPE`
상대가 연결을 끊은 소켓에 쓰면 발생할 수 있다.  
proxy에서 특히 자주 마주친다.  
"내가 잘못 보냈다"기보다 "상대가 먼저 떠났다"는 상황으로 이해하면 된다.

### 자주 헷갈리는 지점
- `fork()` 이후 부모와 자식이 둘 다 같은 코드를 이어서 실행한다는 점을 놓치는 실수
- `execve()`가 새 프로세스를 "추가 생성"한다고 생각하는 실수
- 좀비 프로세스와 고아 프로세스를 혼동하는 실수

### 셀프 체크
1. CGI에서 왜 `fork()` 후 `execve()`를 쓰는가?
2. `dup2()`를 하지 않으면 CGI 출력은 어디로 갈까?
3. 좀비 프로세스는 왜 생기고, 보통 어떻게 정리하는가?

## 3-5. tiny 웹서버 구현 학습자료

### tiny의 전체 흐름
1. 클라이언트 연결 수락
2. 요청 라인 읽기
3. 헤더들 읽기
4. URI 파싱
5. 정적/동적 판단
6. 정적이면 파일 전송
7. 동적이면 CGI 실행
8. 에러 시 적절한 HTTP 에러 응답

### 함수별 역할 이해
#### `main()`
listenfd 생성, 반복적으로 연결 수락

#### `doit()`
요청 전체를 처리하는 중심 함수  
요청 파싱 -> 정적/동적 구분 -> 각각의 서비스 함수 호출

#### `read_requesthdrs()`
헤더를 빈 줄까지 읽고 소비  
필요하면 출력/디버깅

#### `parse_uri()`
정적/동적 여부 판단  
파일명과 CGI 인자 분리

#### `serve_static()`
파일 검사 -> 응답 헤더 작성 -> 파일 전송

#### `get_filetype()`
파일 확장자에 맞는 MIME type 결정

#### `serve_dynamic()`
헤더 전송 -> `fork/execve/dup2`로 CGI 실행

#### `clienterror()`
404, 403, 501 같은 에러 응답 생성

### 구현 체크리스트
- 요청 라인을 제대로 읽는가
- 헤더를 빈 줄까지 소비하는가
- GET 이외 메서드에 대해 적절히 처리하는가
- 정적/동적 구분 기준이 명확한가
- `stat()` 검사 후 권한/파일 종류를 확인하는가
- 동적 요청에서 `QUERY_STRING`을 정확히 넘기는가
- 에러 응답에도 HTTP 형식이 맞는가

### 테스트 체크리스트
#### telnet으로
- 요청 라인/헤더를 직접 보내서 서버 반응 확인
- HTTP 버전 확인
- 빈 줄이 왜 필요한지 체감

#### 브라우저로
- `/home.html` 같은 정적 요청
- `cgi-bin/adder?123&456` 같은 동적 요청
- 없는 파일 요청 시 에러 처리

### 자주 나는 버그
- `\r\n` 처리 누락
- 헤더의 끝 빈 줄을 안 읽고 본문 처리로 넘어감
- CGI 쿼리 문자열 파싱 실수
- `dup2()` 대상 fd를 잘못 지정
- 정적/동적 구분 로직이 URI 형식에 의존하면서 깨짐

### tiny 단계에서 반드시 말할 수 있어야 할 것
1. 요청 하나가 서버 안에서 어떤 함수 순서로 지나가는가
2. 정적과 동적을 어떻게 구분하는가
3. CGI에서 `fork -> dup2 -> execve`가 왜 필요한가
4. `mmap()`이 정적 파일 전송에서 어떤 역할을 하는가

### tiny_note에 남길 내용
- 정적 콘텐츠 처리 흐름
- 동적 콘텐츠 처리 흐름
- CGI 환경변수와 fd 리다이렉트 이해
- 직접 만든 에러 케이스와 원인

## 4-1. 프록시 서버 개념

### 1) 프록시는 어디에 있는가
클라이언트와 원서버 사이에 있다.

`브라우저 -> 프록시 -> 원서버`

즉, 프록시는
- 클라이언트에게는 서버처럼 보이고
- 원서버에게는 클라이언트처럼 동작한다

### 2) 프록시가 하는 일
- 요청을 읽는다
- 필요한 형태로 요청을 재작성한다
- 원서버에 전달한다
- 원서버 응답을 받아 클라이언트에 전달한다
- 필요하면 캐시에 저장한다

### 3) tiny와 무엇이 다른가
tiny는 **직접 콘텐츠를 제공하는 서버**다.  
proxy는 **다른 서버에게 대신 요청해 주는 중간자**다.

### 4) 요청 재작성
교육용 proxy에서 자주 다루는 포인트:
- 절대 URI 파싱
- Host 헤더 정리
- User-Agent/Connection/Proxy-Connection 처리
- HTTP/1.0 형태로 단순화

핵심은 "브라우저가 보낸 요청을 원서버가 이해하기 쉬운 형태로 다시 만드는 것"이다.

### 자주 헷갈리는 지점
- proxy가 "캐시 서버"와 동의어라고 생각하는 실수
- 브라우저 요청을 그대로 원서버에 흘리면 항상 된다고 생각하는 실수
- proxy가 한쪽에서는 서버, 다른 쪽에서는 클라이언트라는 관점을 놓치는 실수

### 셀프 체크
1. proxy는 클라이언트와 원서버 사이에서 각각 어떤 역할을 하는가?
2. 왜 요청 재작성이 필요한가?
3. tiny와 proxy의 가장 큰 차이는 무엇인가?

## 4-2. 동시성 프로그래밍

### 1) 왜 proxy에서 동시성이 중요해지나
순차 proxy는 한 요청을 처리하는 동안  
다른 요청이 전부 기다려야 한다.

브라우저는 한 페이지를 열 때
HTML, CSS, JS, 이미지 등을 동시에 많이 요청한다.  
그래서 proxy는 동시성을 넣지 않으면 체감 성능이 매우 나쁘다.

### 2) 스레드
스레드는 같은 프로세스 안에서
- 주소 공간을 공유하고
- 실행 흐름만 따로 가진다

proxy에서는 보통
- 연결 하나당 스레드 하나
- 스레드를 detached 상태로 운영

같은 패턴을 자주 쓴다.

### 3) mutex
공유 데이터에 한 번에 한 스레드만 들어가게 하는 가장 기본 락이다.

### 4) semaphore
동시 접근 수를 정교하게 조절할 수 있다.  
reader-writer 구현에서 자주 쓰인다.

### 5) Reader-Writer 문제
캐시는 읽기가 훨씬 많고 쓰기는 적다.  
이때 단순 mutex로 전부 막으면 병렬성이 너무 줄어든다.

reader-writer 설계는:
- 여러 reader는 동시에 허용
- writer는 단독 접근

이 원리에 기반한다.

### 자주 헷갈리는 지점
- "스레드가 많으면 무조건 빠르다"는 오해
- 공유 변수만 위험하고 공유 구조체/배열은 안 위험하다고 생각하는 실수
- detached thread와 join의 차이를 놓치는 실수

### 셀프 체크
1. proxy에서 iterative 구조가 왜 부족한가?
2. mutex만으로도 동기화는 되는데 reader-writer를 쓰는 이유는 무엇인가?
3. detached thread를 쓰는 이유는 무엇인가?

## 4-3. 캐싱 전략

### 1) 캐시가 필요한 이유
같은 객체를 여러 번 요청할 때
원서버까지 다시 가지 않으면 더 빠르다.

### 2) 캐시 설계에서 보는 것
- **객체 크기 제한**: 객체 하나가 너무 크면 저장하지 않음
- **총 캐시 크기 제한**: 전체 메모리 한도를 넘기지 않음
- **교체 정책**: 꽉 차면 무엇을 버릴지 결정

### 3) LRU
가장 오래 안 쓰인 항목을 버리는 방식이다.  
proxy lab에서 매우 자연스럽게 떠오르는 정책이다.

### 4) 동시 접근에서 조심할 점
캐시 조회 중 다른 스레드가 같은 항목을 수정/삭제하면 문제다.  
그래서 캐시는 그냥 배열이 아니라 **동기화 대상인 공유 자원**이다.

### 5) 왜 reader-writer가 잘 맞나
캐시에서는 보통
- 조회(read)가 많고
- 삽입/교체(write)는 적다

그래서 여러 조회를 동시에 허용하고  
삽입/교체 순간만 배타적으로 막는 편이 성능상 유리하다.

### 셀프 체크
1. 캐시에 객체 크기 제한이 왜 필요한가?
2. LRU는 무엇을 기준으로 버리는가?
3. 캐시에는 왜 reader-writer 락이 잘 맞는가?

## 4-4. proxy 서버 구현 학습자료

### 1단계 - 순차 처리
가장 먼저 해야 할 일은 **기능이 맞는 단일 스레드 proxy**다.

흐름:
1. 브라우저 연결 수락
2. 요청 읽기
3. URI 파싱
4. 원서버 연결
5. 재작성한 요청 전송
6. 원서버 응답 수신
7. 클라이언트에 전달
8. 종료

이 단계에서 캐시나 스레드는 아직 신경 쓰지 않는다.

### 2단계 - 병렬 처리
연결 하나를 받으면 스레드를 만들고  
그 스레드가 해당 클라이언트를 끝까지 처리하게 한다.

포인트:
- main thread는 accept 루프 유지
- worker thread는 connfd 소유권을 가져감
- fd 전달과 close 책임을 명확히 해야 함

### 3단계 - 캐시 추가
응답을 전달하면서
- 캐시에 넣을 수 있는 크기인지 판단
- 넣을 수 있으면 저장
- 다음 요청에서 캐시 hit면 원서버로 안 감

### 구현 체크리스트
- 요청 라인을 제대로 파싱하는가
- 호스트/포트/경로를 분리할 수 있는가
- 브라우저 헤더를 어떤 기준으로 유지/대체하는가
- Connection 관련 헤더를 적절히 다루는가
- 원서버 응답을 끝까지 relay하는가
- 캐시 hit/miss 흐름이 분리되어 있는가
- 캐시 접근에 락이 걸려 있는가

### 자주 나는 버그
- 절대 URI 파싱 오류
- Host 헤더가 없거나 중복됨
- 스레드가 받은 connfd 포인터를 잘못 공유해서 경쟁 상태 발생
- 응답 전체를 읽기 전에 종료
- 캐시 삽입 중 동기화 누락
- LRU 갱신 시점이 불명확함

### proxy 단계에서 반드시 말할 수 있어야 할 것
1. 프록시가 양쪽에서 각각 어떤 역할을 하는가
2. 왜 요청 재작성이 필요한가
3. 순차 처리와 병렬 처리의 차이가 어디서 생기는가
4. 캐시에서 왜 reader-writer 락이 더 적합한가

### proxy_note에 남길 내용
- 요청 재작성 규칙
- 스레드 모델과 fd 소유권 규칙
- 캐시 hit/miss/eviction 흐름
- 실제로 난 race condition 혹은 deadlock 위험 지점

## 4-5. 테스트의 의미 이해하기

### 1) basic test
기본 기능이 되는지 본다.

검증 포인트:
- 요청을 제대로 전달하는가
- 응답을 제대로 relay하는가
- 헤더 처리가 최소 요구를 만족하는가

### 2) concurrency test
동시 요청을 처리할 수 있는지 본다.

검증 포인트:
- 여러 요청이 겹쳐도 멈추지 않는가
- 스레드 모델이 실제로 병렬성을 주는가
- fd/메모리 관리가 안정적인가

### 3) cache test
캐시가 기능하고 동기화가 맞는지 본다.

검증 포인트:
- 같은 객체를 재사용하는가
- 크기 제한을 지키는가
- 캐시 접근 경쟁 상태가 없는가

### 반드시 스스로 설명해 봐야 하는 질문
- basic는 "기능 맞음"을 확인한다.
- concurrency는 "여러 요청이 겹쳐도 구조가 안전함"을 확인한다.
- cache는 "성능 최적화가 논리적으로도 안전함"을 확인한다.

## 5. 구현 전에 외워 둘 핵심 흐름 요약

### echo
`socket -> bind -> listen -> accept -> echo -> close(connfd)`

### tiny
`accept -> read request line -> read headers -> parse uri -> static or dynamic -> response`

### CGI
`fork -> set env -> dup2(connfd, STDOUT) -> execve`

### proxy
`accept -> parse request -> connect to end server -> forward request -> relay response -> optional cache`

### 캐시 동기화
`lookup(read lock) -> hit면 반환 / miss면 서버에서 가져오기 -> insert(write lock)`

## 6. 단계별 "막히면 여기로 돌아오기"

### 1단계에서 막힌다
- server/client 시스템콜 순서를 말로 다시 써 본다.
- listenfd와 connfd 역할을 다시 구분한다.
- short count와 RIO를 다시 본다.

### echo에서 막힌다
- connfd에 read/write하는지 확인한다.
- accept 이후 fd가 새로 생긴다는 점을 다시 본다.
- telnet으로 수동 테스트한다.

### tiny에서 막힌다
- HTTP 요청 형식을 직접 손으로 써 본다.
- 요청 라인과 헤더의 경계를 다시 본다.
- 정적/동적 구분 기준을 함수 단위로 다시 적는다.

### CGI에서 막힌다
- 부모/자식이 각각 무슨 일을 하는지 분리해서 적는다.
- `dup2()`가 "표준출력의 목적지를 바꾸는 것"임을 다시 본다.
- QUERY_STRING이 어디서 오는지 확인한다.

### proxy에서 막힌다
- 먼저 single-thread 기능부터 맞춘다.
- 그 다음 thread를 넣는다.
- 마지막으로 cache를 넣는다.
- 한꺼번에 다 넣으려 하지 않는다.
